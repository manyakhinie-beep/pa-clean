# PACKAGING_PLAN — Упаковка `pa-clean` в десктоп-приложение

> Исследование: как поставлять pa-clean конечному пользователю как готовое
> приложение, не требующее ручной установки Python / uv / зависимостей и
> **не требующее root-прав**.
>
> Основной кандидат — [PyApp](https://ofek.dev/pyapp/latest/).
> Альтернативы рассмотрены ниже с трейдоффами.

---

## Контекст

Сейчас (май 2026) поставка `pa-clean` устроена так:

1. Пользователь клонирует репозиторий.
2. Запускает `make.sh` → `uv` ставит Python 3.11+ + зависимости.
3. Конфигурирует `.env`, запускает `pa serve`.
4. Открывает WebUI в браузере.

Это требует от пользователя: git, терминал, понимание venv,
аккуратность с правами доступа Mail.app / Calendar.app. Цель —
схлопнуть это в «скачал `.app`, перетащил в `~/Applications`,
запустил». При этом MLX и AppleScript должны продолжать работать.

**Ограничения проекта**, которые задают выбор:

- MLX-LM работает **только** на macOS Apple Silicon (arm64) — никаких
  кросс-платформенных одиночных бинарей не получится; пакет распространяется
  только для `darwin-arm64`.
- Модель GigaChat3.1-10B-A1.8B-MLX-4bit весит ~6 GB; bge-m3 — ~2 GB. Их
  нельзя класть внутрь `.app` — это сделает бандл огромным и сломает
  Gatekeeper кеширование. Веса должны качаться при первом запуске.
- AppleScript к Mail/Calendar требует разрешения TCC (Privacy →
  Automation). Запрос всплывает при первом вызове `osascript` — это
  стандартный механизм macOS, ничего обходить не нужно.
- Никакой root: пользователь должен иметь возможность поставить
  приложение в `~/Applications` (свою home-папку), без `sudo`.

## Что такое PyApp

PyApp — это **Rust-обёртка-загрузчик** (~5 MB бинарник), которая при
первом запуске:

1. Скачивает изолированный CPython-дистрибутив
   ([python-build-standalone](https://github.com/astral-sh/python-build-standalone))
   в `~/Library/Application Support/pyapp/<app-id>/`.
2. Создаёт виртуальное окружение, ставит Python-зависимости
   (из embedded wheelhouse или PyPI).
3. Запускает указанную в config entry-point (например `personal_assistant.cli:main`).

После первого запуска весь Python живёт в `Application Support`, бинарник
PyApp работает как лаунчер. Обновления — встроенный механизм `self update`
(сравнивает версию пакета, обновляет venv).

Документация: https://ofek.dev/pyapp/latest/

## План PyApp для pa-clean

### 1. Зависимости (что добавляется)

**Build-time** (на машине разработчика / CI):
- Rust toolchain (`rustup`, `cargo`) ≥ 1.75 — для сборки PyApp бинаря.
  `cargo install pyapp` ставит сам launcher как cargo-крэйт.
- `hatch` или `uv build` — собрать pa-clean в `.whl` (на этом уровне он
  уже подсчитан как dist-артефакт).
- Опционально: `create-dmg` или `dmgbuild` — собрать `.dmg` для удобной
  раздачи.
- Apple Developer ID (≈ $99/год) — нужен только если хочется обойти
  Gatekeeper «приложение от неизвестного разработчика». Без подписи
  пользователь делает Ctrl-клик → «Открыть» один раз; это допустимо для
  внутреннего корпоративного использования.

**Run-time** (на машине пользователя):
- Никаких **новых** зависимостей сверх того, что уже нужно сейчас.
  PyApp сам подкладывает Python-runtime; pip-зависимости pa-clean
  (`fastapi`, `mlx-lm`, `sentence-transformers`, etc.) ставятся из
  embedded wheelhouse или PyPI.
- ~150 MB на CPython-distribution в `Application Support`.
- ~3-5 GB на pip-кэш зависимостей (mlx, torch — нет, sentence-transformers
  тянет ONNX runtime).
- ~6-8 GB на веса моделей (MLX + embedding) — качаются отдельно при
  первой настройке.

**Итого первый запуск**: пользователь видит индикатор «Устанавливаем
ассистента…», ждёт 3-8 минут (зависит от сети), потом ассистент готов.

### 2. UX установки

```
1. Пользователь скачивает PaClean-1.0.0-arm64.dmg (или .app в zip)
   с GitHub Releases или внутреннего портала.
2. Перетаскивает PaClean.app в ~/Applications (НЕ /Applications —
   избегаем sudo). При желании можно и в /Applications, если есть
   admin-права — но это не обязательно.
3. Первый запуск:
   3a. macOS просит подтвердить «приложение от неизвестного разработчика»
       (если бандл не подписан Developer ID). Ctrl-клик → Открыть → один
       раз; дальше нет.
   3b. PyApp-лаунчер открывает окно «Устанавливаем ассистента» —
       прогрессбар. Качает CPython (~150 MB), ставит pip-зависимости
       (~500 MB).
   3c. После завершения автоматически открывается локальный WebUI в
       браузере по умолчанию.
4. Пользователь идёт в WebUI → вкладка «Настройка» → жмёт «Скачать
   модель» — фоновая загрузка ~6 GB MLX-весов в ~/.cache/huggingface.
5. По мере первого использования Mail/Calendar функций macOS показывает
   стандартные диалоги «PaClean хочет управлять Mail» — пользователь
   соглашается. Никакого ручного `tccutil` не требуется.
6. (Опционально) Login Item: галочка «Запускать при входе» в Настройках
   ассистента — через Apple-овский ServiceManagement API,
   не требует прав.
```

**Сравнение с текущим UX:** сейчас у пользователя ~12 шагов с терминалом,
требуется понимание .env, git, uv. После — 2 действия мыши + ожидание.

### 3. Бенефиты

- **Один артефакт.** GitHub Release с `PaClean-1.0.0.dmg`, и больше
  ничего не нужно. Никаких README с командами.
- **Изоляция Python.** Системный Python пользователя не трогается.
  Если у него /usr/local/bin/python3 от Homebrew с непонятным состоянием —
  pa-clean это игнорирует, у нас свой.
- **Воспроизводимость.** Все пользователи получают **тот же**
  python-build-standalone CPython той же версии. Конец «у меня заработало,
  у тебя — нет».
- **Самообновление.** PyApp поддерживает `pa update` (или, если делать
  GUI, кнопка «Проверить обновления») — сравнивает версию через
  PyPI/GitHub, тянет новый .whl, ставит. **Не пересобирает Python**,
  только обновляет наш пакет — секунды.
- **Размер бинаря.** 5-8 MB launcher против 200+ MB у PyInstaller-однофайла.
  CPython и зависимости качаются **один раз** и переиспользуются между
  обновлениями.
- **Не требует root.** Установка в `~/Applications` стандартная. Запись
  во `~/Library/Application Support/` — без прав.
- **Локально, без облака.** Веса моделей, vault, конфиг — всё в home
  пользователя; pa-clean не делает сетевых вызовов кроме MLX-loader-а и
  обновления самого пакета (если включить эту опцию).

### 4. Скорость и стабильность

**Скорость:**
- **Первый запуск** (cold start): 3-8 минут на стандартной сети — это
  лимитировано пропускной способностью PyPI/HuggingFace. На корпоративной
  сети с прокси может быть медленнее; рекомендуется опция embedded
  wheelhouse (см. ниже).
- **Все последующие запуски**: 1-3 секунды до готовности FastAPI.
  PyApp launcher практически мгновенный (Rust), `uv venv activate` уже
  закэширован, импорт fastapi+mlx-lm — основное время.
- **MLX inference**: вообще не зависит от PyApp — это runtime-вопрос.
  Скорость как при ручной установке.

**Стабильность:**
- PyApp в проде с 2023 года. Используется в `hatch`, `posit-dev/Connect`,
  ряде корпоративных решений. Активно maintained (ofek — core dev hatch).
- Главные риски:
  1. **PyPI-сетевые сбои** при первом запуске — PyApp пытается
     повторить, но если корпоративный прокси режет — bootstrap падает.
     Решение: **offline mode** через embedded wheelhouse (см. ниже).
  2. **MLX-несовместимость**: если выходит mlx-lm с breaking-change,
     самообновление сломает работу пользователя без отката. Решение:
     pin версий в pyproject.toml и semver-теги релизов.
  3. **macOS upgrade**: новый macOS может сломать python-build-standalone
     (редко, но было). Lag между релизом macOS и обновлением PBS — дни.
     Решение: проверять beta-macOS в CI, держать запас pinned-версии PBS.

**Embedded wheelhouse:** PyApp поддерживает запекание всех wheels
в бинарь (`PYAPP_PROJECT_DEPENDENCY_FILE=requirements.lock`). Тогда
bootstrap **не лазает в PyPI** — все зависимости извлекаются из самого
бинаря. Размер вырастает до 200-400 MB, но первый запуск работает
оффлайн. Для корпоративного развёртывания это правильный выбор.

## Альтернативы

Все без root-прав, все ставятся в `~/Applications`.

### Альт. 1: py2app (классический способ macOS)

**Что:** Python-инструмент (`pip install py2app`), генерит `.app` бандл
с встроенным CPython + всеми зависимостями. Чисто Python-нативный,
никакого Rust.

- ✅ Самый «маковский» — настоящий `.app`-бандл с Info.plist, иконкой,
  правильным launchd-поведением. Гладко интегрируется с Login Items,
  Dock, Spotlight.
- ✅ Не требует сети при первом запуске — всё внутри бандла.
- ✅ Есть встроенная поддержка bundle-resources, plist-кастомизации.
- ⚠ **Бандл огромный**: 300-800 MB только за счёт sentence-transformers,
  pydantic, fastapi. Веса моделей — отдельно.
- ⚠ **Самообновление вручную** — нужно скачать новый .dmg. Нет
  встроенного `--update`.
- ⚠ Капризен в настройке: рецепты под mlx-lm придётся писать самим,
  py2app часто пропускает динамические импорты.
- ⚠ Каждый Python-релиз требует обновления py2app — иногда лагает.

**Хорошо подходит, если:** нужен «настоящий» .app для App Store / Apple
Notarization / корпоративного MDM. Не самый удобный для итеративной разработки.

### Альт. 2: PyInstaller

- ✅ Самый зрелый, кросс-платформенный.
- ✅ Один-файл-бинарь (`--onefile`) — drag-and-drop.
- ⚠ Бандл 200-400 MB, медленный cold-start (1-3s на распаковку tmp).
- ⚠ MLX часто требует ручных hidden-imports.
- ⚠ Антивирусы любят пометить onefile-бинарь как подозрительный —
  на macOS это менее остро, чем на Windows, но Gatekeeper всё равно
  бубнит.
- ⚠ Нет самообновления.

**Хорошо подходит, если:** нужно поддерживать сразу macOS+Windows+Linux
в одном workflow. Для исключительно macOS-проекта избыточен.

### Альт. 3: Briefcase (BeeWare)

- ✅ Спроектирован для производства **настоящих** macOS .app-бандлов
  с native-feel (Cocoa, нормальный launchd).
- ✅ Templates для notarization, code-signing, dmg-сборки.
- ✅ Управляет ресурсами через `pyproject.toml`.
- ⚠ Bundle 300-600 MB.
- ⚠ Briefcase ориентирован на GUI-приложения (Toga), у нас сервер +
  WebUI в браузере — overhead конфигурации.
- ⚠ Кривая обучения чуть круче, чем py2app.

**Хорошо подходит, если:** в будущем планируется native-GUI (нативные
окна macOS). Если так и останется веб-интерфейс — переплата.

### Альт. 4: Nuitka

- ✅ Компилирует Python в C → нативный исполняемый код.
- ✅ Самый компактный результат (40-80 MB при хорошей конфигурации).
- ✅ Чуть быстрее cold-start.
- ⚠ Долгая сборка (10-30 минут vs секунды у PyApp).
- ⚠ Не все Python-фичи поддержаны идеально; mlx-lm может потребовать
  отдельных воркараундов.
- ⚠ Платный enterprise-вариант с лучшей оптимизацией; community-версия
  достаточно, но требует ручной настройки.
- ⚠ Нет встроенного self-update.

**Хорошо подходит, если:** нужна максимальная защита кода от обратной
инженерии (Nuitka компилирует, не упаковывает). Для open-source pa-clean
смысла не имеет.

### Альт. 5: shiv / pex (zip-app)

- ✅ Один-файловый zip-archive с Python-зависимостями. Очень компактно
  (~5-15 MB).
- ⚠ **Требует системный Python ≥ 3.11**. Для нас это deal-breaker:
  у пользователя может не быть Python, или быть «не тот».
- ⚠ Не есть .app-бандл — нужно отдельно оборачивать.

**Хорошо подходит, если:** аудитория — Python-разработчики (CLI-тулы для
inner-tooling). Для конечного пользователя — нет.

### Альт. 6: Tauri + Python sidecar

- ✅ Современный Rust-фреймворк для desktop-апп. WebUI как Native окно
  (WKWebView), Python как embedded sidecar-процесс.
- ✅ ~20-40 MB бандл.
- ✅ Очень нативный feel — кнопки macOS, меню, шорткаты.
- ⚠ Сложно: нужен Rust + frontend (React/Vue/Svelte) + Python-инкорпорация.
  Это +2-4 недели работы только на инфраструктуру.
- ⚠ Дублирование: у нас уже есть WebUI на vanilla-JS — пришлось бы либо
  его переписывать, либо обёртывать iframe.

**Хорошо подходит, если:** нужен native-feel-приложение с собственным
окном. Для full-stack-redesign — много.

### Альт. 7: pkg-installer (без app-бандла)

- ✅ Простой `.pkg` (Apple Installer), который кладёт всё в
  `~/Library/PaClean/` и регистрирует launchd-сервис.
- ✅ Без root, если пользователь ставит «just for me».
- ⚠ Нет иконки в Dock, нет тогглеров. Только CLI + WebUI.
- ⚠ Менее интуитивно: пользователь не понимает, как удалить.

**Хорошо подходит, если:** аудитория — IT-специалисты в корпоративной
среде, где MDM раскатает .pkg на все маки автоматически.

## Bootstrap MLX-весов (общая часть для всех вариантов)

Модельные веса (6-12 GB) **никогда не упаковываются в бандл**, независимо
от выбора инструмента. Шаги бутстрэпа модели:

1. При первом запуске WebUI открывается вкладка **Настройка → Модель**.
2. Кнопка «Скачать модель GigaChat3.1-10B-A1.8B-MLX-4bit» запускает фоновую
   задачу: `huggingface_hub.snapshot_download(repo_id, cache_dir=~/.cache/huggingface)`.
3. Прогресс отображается в UI; задача переживает перезапуск приложения
   (sync_state.json с watermark).
4. После завершения — `mlx_engine.reload()` подхватывает новый путь.

Это та же логика, что и сейчас в Rules → MLX, разница только в
автозапуске мастера при отсутствии скачанной модели.

## Подпись и distribution

**Минимальный сценарий (внутреннее использование):**
- Не подписывать. Пользователь делает Ctrl-клик → «Открыть» → «Открыть
  всё равно». Один раз; macOS запоминает.

**Корпоративный сценарий:**
- Apple Developer ID (~$99/год) → `codesign --sign "Developer ID Application: …"`.
- Notarization через `notarytool` — Apple сканирует на malware (5-15 минут).
- После notarization Gatekeeper пропускает без warnings.

**Распространение:**
- GitHub Releases — простейший вариант. PyApp поддерживает self-update
  через GitHub Release feed.
- Внутренний artifact registry (jFrog/Nexus) для корпоративного варианта.
- Sparkle framework для расширенного auto-update с changelog'ами —
  если хочется premium UX (не обязательно для PyApp, у него свой механизм).

## Рекомендация

**Берём PyApp** как основной вариант. Конкретно:

1. **Phase 1 (1-2 дня):** PR с `pa-app/Cargo.toml`, `pyapp.toml`, скрипт
   `scripts/build_app.sh`, который собирает arm64 .app для текущей версии.
   Embedded wheelhouse для оффлайн-bootstrap.
2. **Phase 2 (день):** Wizard «Скачать модель» в WebUI — отдельная
   вкладка с прогрессбаром; пользователь больше не лазает в Rules.
3. **Phase 3 (опционально, день):** Подписание Developer ID + notarization
   workflow в GitHub Actions. Решение зависит от того, будет ли проект
   распространяться публично или только внутри организации.

**Почему PyApp, а не py2app/Briefcase:** для pa-clean **главное** —
скорость **итераций релизов**. PyApp = «обновил .whl, запушил тег, юзер
жмёт «Проверить обновления», получает свежий pa-clean за 5 секунд».
py2app требовал бы пересборки 500-мегабайтного .dmg на каждый патч-релиз.

**Когда стоит передумать на Briefcase:** если в будущем pa-clean получит
native-окно вместо WebUI (например, Cocoa-меню с быстрыми действиями).
Тогда Briefcase даёт более правильный фундамент.

**Контр-аргумент против всех вариантов выше — оставить как есть:** если
пользовательская база — это 10-20 разработчиков внутри одной команды,
текущий `make.sh` достаточен. Упаковка имеет смысл при расширении
аудитории до не-инженеров.
