# LoRA fine-tuning из vault-данных

> Дообучаем базовую модель (GigaChat3.1-10B-A1.8B-MLX-4bit) на твоей **реальной**
> переписке, чтобы драфты ассистента звучали как ты, а не как обобщённый
> деловой стиль из коробки.

---

## Идея

В твоём vault уже лежат сотни/тысячи писем: входящее → твой реальный ответ.
Это **золотой стандарт** того, как ты сам пишешь — давайте передадим эту
информацию модели через LoRA-fine-tune.

LoRA (Low-Rank Adaptation) обучает только маленький адаптер (~10-50 MB
вместо ~6 GB полной модели). Запускается на M-чипе через `mlx_lm.lora`,
полный цикл — 10-60 минут.

---

## Pipeline

```
vault → pa lora prepare → data/lora/draft/{train,valid}.jsonl
                      → pa lora train  → data/lora/draft/adapter/
                      → pa lora status → отчёт о текущем адаптере
                      → pa lora clear  → откат к базовой модели
```

---

## Шаг 1 — подготовь датасет

```bash
# Обязательное предусловие: vault синхронизирован и PA_USER_EMAIL задан в .env
pa sync-mail
echo 'PA_USER_EMAIL=igor@example.com' >> .env

# Извлекаем пары входящее → твой ответ
pa lora prepare --task draft
```

Что произойдёт:
1. `lora_dataset.build_pairs` пройдёт по `vault/mail/`, сгруппирует письма
   по `thread_id`.
2. Для каждого твоего ответа найдёт предыдущее **чужое** письмо в треде.
3. Отфильтрует короткие ответы (`< 80 chars` — «Спасибо!», «OK»).
4. Уберёт quoted-history и подписи из тел.
5. Запишет `data/lora/draft/train.jsonl` (80%) + `valid.jsonl` (20%) +
   `manifest.json` с метаданными.

Опции:
```
--max-examples 500          # кап (берём самые свежие)
--min-reply-chars 100       # фильтр коротких ответов
--train-split 0.85          # доля train
--user-email <override>     # перебить PA_USER_EMAIL
```

**Что хорошо, что плохо:**
- ✅ Если у тебя 200-2000 пар — идеально для LoRA на 600 итераций.
- ⚠ < 100 пар — модель скорее всего не научится; стоит синхронизировать
  больший vault (например, `PA_MAIL_DAYS_BACK=365`) или собирать с
  нескольких аккаунтов.
- ❌ > 5000 пар — обучение долгое, скорее всего перебор; `--max-examples 1500`.

---

## Шаг 2 — обучи адаптер

```bash
pa lora train --task draft
```

Что произойдёт:
1. `lora_trainer.run_training` запустит `python -m mlx_lm.lora --train ...`
   как подпроцесс.
2. Прогресс будет печататься в твой терминал в реальном времени:
   `Iter 50: Train loss 1.234, Val loss 1.456, Learning rate 1e-05`.
3. Каждые 100 итераций — checkpoint адаптера.
4. По завершении — `adapter.safetensors` + `adapter_config.json` в
   `data/lora/draft/adapter/`.
5. Полный лог в `data/lora/draft/last_run.log`.

Опции:
```
--iters 600              # количество итераций (200-1500 — типовой диапазон)
--layers 8               # сколько последних слоёв адаптировать (4-16)
--batch-size 1           # на 32GB машине >1 для 10B модели не влезает
--learning-rate 1e-5     # дефолт от MLX team
--model <path>           # перебить PA_MLX_MODEL_PATH
```

**Время обучения** (на M3 Max 36GB, 1000 пар, 600 iters, 8 layers):
~30-45 минут. На M2 8GB — не пытайся, OOM.

**Что смотреть в логе:**
- `Train loss` должен **падать** от итерации к итерации (с 2.0 → к 0.5-1.0).
- `Val loss` должен **тоже** падать, но медленнее.
- Если `Val loss` начал **расти** при падающем `Train` — overfit, надо
  остановить и взять предыдущий checkpoint.

---

## Шаг 3 — проверь что получилось

```bash
pa lora status --task draft
```

Покажет:
- Путь к адаптеру
- Есть ли weights и config
- Размер адаптера в MB
- Метаданные обучения (когда обучен, на скольких парах, какие гиперы)

---

## Шаг 4 — попробуй адаптер вручную

В этой фазе автоматическое применение к live MLX-engine ещё не реализовано
(Phase 2). Использовать адаптер можно через `mlx_lm.generate` напрямую:

```bash
python -m mlx_lm.generate \
    --model "$PA_MLX_MODEL_PATH" \
    --adapter-path data/lora/draft/adapter \
    --prompt 'Письмо от Иванов И.И.:

Тема: Договор на 2026

Прошу прислать договор до пятницы.' \
    --max-tokens 400
```

Сравни выход с тем же промптом **без** `--adapter-path` — это и есть
эффект LoRA.

---

## Шаг 5 — откат если не понравилось

```bash
pa lora clear --task draft
```

Удалит `data/lora/draft/adapter/`. Датасет (`train/valid.jsonl`) **не**
трогает — можно переобучить с другими гиперпараметрами.

---

## Типичные настройки гиперов

Для GigaChat3.1-10B-A1.8B на ~1000 деловых писем:

| Гипер | Значение | Когда менять |
|-------|----------|--------------|
| `--iters` | 600 | Меньше пар → меньше iters (200-400); больше → больше (800-1200) |
| `--layers` | 8 | Если loss не падает — увеличить до 12-16 |
| `--batch-size` | 1 | На M-чипах с ≥ 64GB можно поднять до 2-4 |
| `--learning-rate` | 1e-5 | Если loss скачет — снизить до 5e-6 |

Если первый запуск дал плохое качество (драфты звучат не как ты) —
попробуй в этом порядке:
1. Увеличь `--iters` до 1000.
2. Увеличь `--layers` до 12.
3. Собери больше пар: `pa sync-mail` с большим `PA_MAIL_DAYS_BACK`,
   потом снова `pa lora prepare`.

## Безопасность данных

- **Всё локально.** Ни одного вызова в OpenAI/HuggingFace/etc. во время
  обучения. `mlx_lm.lora` работает с локально скачанной моделью + локальным
  vault.
- Адаптер хранится в `data/lora/draft/adapter/` — не уходит за пределы
  твоей машины. Бэкап делай руками если ценен.
- Не пересылай адаптер коллегам как есть — в его весах **закодирован**
  твой стиль переписки на конкретных кейсах. Маленький leak privacy.

---

## Что НЕ реализовано в Phase 1 (для Phase 2)

- **Hot-swap адаптера в running engine** — сейчас MLX-engine pa-clean не
  знает про `data/lora/draft/adapter/`. Чтобы драфты в WebUI пользовались
  адаптером, нужно отдельно перезагрузить engine с `--adapter-path` (это
  отдельная задача).
- **UI кнопки** «Обучить мой стиль» в Правилах.
- **Прогресс обучения через WebUI** (SSE-stream loss/iters в реальном
  времени).
- **A/B-сравнение** генераций до/после адаптера.
- **Поддержка task=summarize, task=delegate** — сейчас только `draft`.

Эти задачи — Phase 2. Phase 1 даёт CLI-пайплайн, на котором можно
проводить эксперименты руками.

## Откуда брать ещё качество

Если LoRA на собственных письмах не даёт нужного эффекта:

1. **Качество данных > количество.** Лучше 200 «хороших» пар (длинные
   развёрнутые ответы) чем 2000 односложных. Подними `--min-reply-chars`
   до 200-300, посмотри сколько пар останется.
2. **Чистка артефактов.** В `_clean_body` сейчас режутся только Quoted-
   history + sigdash подпись. Если в твоей почте много auto-replies
   ("Out of office..."), добавь свой фильтр в `lora_dataset.py`.
3. **Curated dataset.** Прежде чем `pa lora train`, открой
   `data/lora/draft/train.jsonl` и **руками** удали несколько самых
   неудачных примеров (один редко-странный ответ может перекосить всю
   модель). 5-минутная ручная чистка ценнее +200 итераций.
4. **Domain adaptation poверх LoRA.** Если хочется не просто
   персональный стиль, а доменный — например «юридические письма
   только», — собери subset из vault через grep по ключевым словам
   и подай в `--data` напрямую.
