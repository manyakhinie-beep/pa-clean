# PILOT_DISTRIBUTION — Доставка `pa-clean` тестировщикам-не-инженерам

> Тактический план: как **на этой неделе** дать 5-10 коллегам-не-инженерам
> с ограниченными правами macOS попробовать pa-clean, собрать обратную
> связь и итеративно улучшать.
>
> Связан с [PACKAGING_PLAN.md](PACKAGING_PLAN.md) — там стратегический
> взгляд на продакшен-упаковку. Здесь — пилот.

---

## Профиль тестировщика

Что мы про него знаем (и что меняет дизайн):

- Не знает что такое терминал, git, Python.
- macOS под управлением MDM (Jamf / Intune / Kandji) с включённым
  Gatekeeper. Возможно, FileVault, EDR (CrowdStrike, SentinelOne).
- **Нет** admin-прав. `sudo` не работает.
- `/Applications` может быть запрещён для записи; `~/Applications`
  обычно открыт.
- PyPI и HuggingFace **могут** быть заблокированы корпоративным прокси —
  планируйте офлайн-bootstrap.
- Mail.app и Calendar.app настроены (есть рабочая почта), но любые
  запросы автоматизации пользователь видит впервые в жизни.
- Скорее всего на 16-32 GB RAM и Apple Silicon — это норма для нашего
  типового парка.

Главное правило: **если на каком-то шаге тестировщик «не понял что
делать» — это наш баг**, а не его. Все шаги должны быть нажатиями
мыши с понятным текстом, или вообще автоматически.

---

## Что должен сделать тестировщик (target UX)

```
1. Получает в Slack ссылку «PaClean v0.9-pilot.zip» или сразу .dmg.
2. Двойной клик. macOS говорит «приложение не от Apple, открыть?»
   Тестировщик жмёт «Открыть».
3. Открывается приветственное окно WebUI в браузере:
   «Привет! Я ваш персональный ассистент. Сейчас установлю себя — это
    займёт пару минут. Ничего нажимать не нужно».
4. Прогрессбар → автоматически открывается главный экран.
5. macOS показывает диалог «PaClean хочет управлять Mail» — тестировщик
   жмёт «OK». В UI крупный шаг: «Нажмите OK в окошке macOS» с картинкой.
6. То же для Calendar.
7. Главный экран говорит «Готов. Попробуйте: Inbox / Чат / Сегодня».
```

Всё. Никаких .env, никакого `chmod`, никаких объяснений про venv.

## 3 главных препятствия (и как их обходим)

### Препятствие 1: Gatekeeper

Симптомы у тестировщика: «приложение «PaClean» нельзя открыть, так как
у него поврежденный код подписи».

Это **не** про malware — это про отсутствие Developer ID. Без подписи
macOS не даёт запустить даже через Ctrl-клик на корпоративных машинах
с MDM-политикой.

**Решение для пилота:**

1. **Apple Developer ID Application certificate** (Apple Developer Program,
   $99/год) — у нас должен быть один такой, прибитый к корпоративному
   аккаунту. Если ещё нет — это **блокер**, заводим за 1-2 дня.
2. Подписываем бандл: `codesign --deep --force --options runtime --sign
   "Developer ID Application: <Company>" PaClean.app`.
3. Нотарирование через `notarytool submit … --wait` — Apple сканирует
   на malware (5-15 минут). После успеха `stapler staple PaClean.app`
   приклеивает «штамп» к бандлу, чтобы Gatekeeper работал офлайн.
4. После этого тестировщик видит максимум «приложение скачано из
   интернета, открыть?» — стандартный диалог, одна кнопка.

**Альтернативный путь без Developer ID** (если certificate ещё не оформлен):
- ad-hoc подпись + Ctrl-клик → «Открыть» один раз.
- На MDM-машинах **скорее всего не сработает** — IT-политика блокирует
  unsigned bundles полностью.
- Делать ad-hoc только для коллег-инженеров «как себе», не для пилота.

### Препятствие 2: Сеть (PyPI / HuggingFace)

Симптомы: тестировщик жмёт «Запустить», приложение зависает на
«Устанавливаем…» с ошибкой timeout.

Причина: корпоративный прокси режет `pypi.org` или `huggingface.co`.
Часто разрешён только внутренний artifact mirror.

**Решение:**

- **Embedded wheelhouse** в PyApp-бандле (см.
  [PACKAGING_PLAN.md](PACKAGING_PLAN.md) → раздел Embedded wheelhouse) —
  все pip-зависимости запекаются в бинарь, bootstrap **не лазает в PyPI**.
  Размер бандла +200-400 MB, но 100% офлайн на этапе установки.
- **Модельные веса в отдельном архиве** — лежат на внутреннем S3 / Nexus
  / SharePoint с подписанной ссылкой. На главном экране кнопка «Скачать
  модель» с прогрессбаром, ссылка ведёт **на внутренний host**, а не на
  HuggingFace.
- Альтернатива (если внутренний mirror недоступен) — **тестовый zip
  с моделью**. Тестировщик качает 6 GB архив, перетаскивает в окно
  ассистента — мы извлекаем в `~/Library/Application Support/PaClean/models/`.
- Если HuggingFace **доступен** через прокси (у разных компаний по-разному),
  можно использовать стандартный snapshot_download — но проверьте это
  на одном тестировщике ДО рассылки всем.

### Препятствие 3: Mail/Calendar Automation permissions

Симптомы: тестировщик жмёт «Sync», получает «0 писем», в логах
`error 1743 not authorized`.

Это **нормально на первом запуске** — macOS требует явного одобрения
автоматизации каждой пары (приложение, целевое приложение). Проблема в
том, что тестировщик может пропустить диалог или нажать «Don't allow».

**Решение в UI** (нужен отдельный onboarding-шаг до первого sync):

1. Главный экран сразу после bootstrap показывает блок «Разрешения»:
   ```
   ✅ Автоматизация Mail.app — нажмите кнопку, появится системное окно,
      выберите «Разрешить».
   ❌ Автоматизация Calendar.app — то же.
   ```
2. Кнопка триггерит «пустой» osascript-вызов (например, `tell application
   "Mail" to return name of first account`) — это всплывает диалог.
3. Цвет индикатора меняется на зелёный когда tccutil подтверждает.
4. Если пользователь случайно нажал «Don't allow», блок показывает
   инструкцию со скриншотом: «Откройте Системные настройки → Конфиденциальность
   → Автоматизация → найдите PaClean → поставьте галочку у Mail».

Это **критичный** момент UX — без него пилот провалится у 80%
тестировщиков, потому что они не поймут что нужно делать.

## Способы доставки бандла (без admin)

В порядке предпочтения:

### 1. Внутренний artifact server / SharePoint / Confluence-attachment

- ✅ Контролируем доступ.
- ✅ Можем подписать ссылку под конкретного пилота.
- ✅ Можем убрать ссылку в любой момент (отзываемая раздача — критично
  для pilot-версии с потенциальными багами).
- ⚠ Нужно настроить storage с поддержкой больших файлов (>200 MB для
  embedded-wheelhouse-варианта).

### 2. Внутренний MDM (Jamf Self Service / Intune)

- ✅ Пользователь не качает руками — приложение появляется в каталоге
  «Self Service».
- ✅ IT-отдел уже одобрил его как корпоративное → Gatekeeper тихий.
- ⚠ Нужно согласование с IT, может занять недели. Не подходит для
  быстрого пилота.

### 3. GitHub Releases (private repo) + Personal Access Token

- ✅ Знакомый процесс для команды разработки.
- ⚠ Тестировщик-не-инженер должен залогиниться на GitHub — лишний шаг.
- ⚠ PAT может быть запрещён корпоративной политикой.

### 4. Direct Slack/Email attachment

- ✅ Самый быстрый канал.
- ⚠ Slack режет файлы > 1 GB, email обычно > 25 MB. Можно положить
  только подписанный launcher (~10 MB), а тяжёлый bootstrap делать
  через WebUI (ссылка на внутренний storage).
- ⚠ EDR-системы любят карантинировать unsigned/новые .dmg из почты.

**Рекомендация:** SharePoint/внутренний-S3 + ссылка в Slack. MDM как
phase 2, когда пилот доказал что продукт работает.

## Сбор обратной связи и логи

Что должно работать **из коробки**, чтобы тестирование было полезным:

### 1. Кнопка «Сообщить о проблеме» в WebUI

Открывает diff-блок:
- Сборка-версия + git SHA приложения (мы знаем что юзер тестит).
- Запрос «опишите проблему» (свободный текст).
- Чекбокс «приложить последние 200 строк логов» (по умолчанию OFF —
  privacy first).
- Кнопка «Скопировать в буфер» — формат markdown, готов к вставке в
  Slack/Linear.

Никакой автоматической отправки — тестировщик контролирует когда и кому
отдаёт информацию.

### 2. Diagnostics-блок в UI

На главной странице небольшая иконка «i» открывает:
- Версия приложения, дата сборки.
- Статус Mail / Calendar permissions (зелёный/красный).
- Статус модели (загружена / не загружена / ошибка).
- Последние 3 ошибки из лога с timestamp-ом.

Это убирает 70% диалогов «у меня не работает, что показать?»

### 3. Telemetry — **только** opt-in

- Окно «Поделиться анонимной телеметрией» с явным списком:
  «версия приложения, время отклика, частота ошибок — без содержимого
  писем, без email-ов, без названий контактов».
- По умолчанию **выключено**.
- Кнопка «Посмотреть что отправляется» открывает JSON-превью.

В рамках пилота можно вообще убрать — пилоты малочисленны, ручной
канал обратной связи лучше масштабируется при N=5-10 человек.

## Hot-fix / итерации без переустановки

Чтобы тестирование было полезным, надо **быстро** правлять баги и
доставлять фикс. Если каждое исправление требует от тестировщика
переустановить 500-мегабайтный бандл — обратная связь умрёт.

Варианты:

### Вариант А: PyApp self-update

PyApp поддерживает `pa update`, который сравнивает версию `.whl` с
PyPI/GitHub и обновляет venv внутри `Application Support/`.

- ✅ Тестировщик жмёт кнопку «Проверить обновления» в UI → 30 секунд →
  готово.
- ✅ Никакого drag-and-drop в Applications.
- ⚠ Требует доступа к серверу обновлений — для пилота можно крутить
  свой простой HTTP-эндпоинт `/updates/latest.json`.

### Вариант Б: Auto-pull последней версии при старте

При запуске приложение ходит на внутренний endpoint и если есть
новее — тихо обновляет venv. Опционально показывает «Обновлено до
v0.9.3».

- ✅ Тестировщик вообще не задумывается об обновлениях.
- ⚠ Требует разрешения «приложение само загружает код» — некоторые
  IT-политики против. Для пилота ОК, для прода — обсудить с InfoSec.

### Вариант В: Patch-bundles вместо полных

Если bug-fix трогает только Python-код (не Rust launcher) — отдаём
маленький `pa-clean-v0.9.3.whl` (~5 MB). Кнопка «Применить патч» в UI
заменяет файл в venv и перезапускает FastAPI-сервер.

Это самый «инженерский» вариант, но даёт самый быстрый цикл обратной
связи.

**Для пилота рекомендуется А** — стандартный механизм PyApp, минимум
кастома.

## Чеклист первого пилота

В таком порядке:

- [ ] **Phase 0 — за неделю до пилота**
  - [ ] Завести Apple Developer ID Application certificate (если ещё нет).
  - [ ] Договориться с IT, какие домены / прокси доступны на машинах
    тестировщиков. Если PyPI/HuggingFace заблокированы — закладываем
    embedded wheelhouse + внутренний model-storage.
  - [ ] Согласовать SharePoint / внутренний S3 для раздачи бандла.

- [ ] **Phase 1 — сборка**
  - [ ] PyApp config: `pyapp.toml` + entry-point на `personal_assistant.cli:main`.
  - [ ] Embedded wheelhouse через `PYAPP_PROJECT_DEPENDENCY_FILE=requirements.lock`.
  - [ ] `scripts/build_pilot.sh` собирает .app, подписывает Developer ID,
    нотарирует, степлит, упаковывает в .dmg.
  - [ ] CI на GitHub Actions с macOS-13-arm64 runner для воспроизводимости.

- [ ] **Phase 2 — onboarding UX**
  - [ ] Welcome-страница в WebUI при отсутствии модели и/или permissions.
  - [ ] Шаг «Разрешения» с триггерами osascript + индикаторами.
  - [ ] Шаг «Скачать модель» с прогрессбаром.
  - [ ] Кнопка «Сообщить о проблеме» в шапке + Diagnostics-блок.

- [ ] **Phase 3 — pilot dry-run (внутри команды разработки)**
  - [ ] Один не-инженер из команды (PM / дизайнер) проходит весь путь
    с нуля, без подсказок. Замеряем где он застрял.
  - [ ] Чиним выявленные проблемы. Без этого шага пилот провалится.

- [ ] **Phase 4 — рассылка пилотам**
  - [ ] 5-10 тестировщиков. Личное сообщение в Slack со ссылкой и
    одним абзацем «что мы хотим узнать».
  - [ ] Календарная встреча через 7 дней на 30 минут для группового
    debrief.
  - [ ] Между этим — открытый канал в Slack для вопросов.

- [ ] **Phase 5 — итерация**
  - [ ] Раз в 2-3 дня собираем фидбек, патчим, рассылаем через
    self-update (PyApp).

## Что **не** делать в пилоте

- Не просить тестировщиков «открыть терминал». Это сразу +50% дроп-аута.
- Не отправлять unsigned бандл с инструкцией «правый клик → Открыть»
  — на корпоративном маке это просто не работает.
- Не давать тестировщикам конфиг через `.env`. Все настройки доступны
  через Rules-вкладку WebUI, и должно быть так.
- Не требовать чтобы тестировщик скачивал модель руками. Wizard
  делает это за один клик.
- Не привязываться к одному не-инженеру в team — найдите хотя бы троих
  из разных подразделений. Маркетинг, продукт, продажи — разные паттерны
  использования покажут разные баги.

## Размер артефакта и время пилота — реалистично

| Артефакт                                            | Размер     |
|------------------------------------------------------|-----------|
| Подписанный .dmg launcher + embedded wheelhouse     | ~400 MB    |
| Модельные веса (GigaChat3.1 + bge-m3)               | ~8 GB      |
| Распакованный bootstrap в Application Support       | ~700 MB    |
| Итого на диске у пилота                              | ~9 GB      |

| Шаг                                            | Время  |
|-------------------------------------------------|--------|
| Скачивание .dmg                                 | 1-3 мин |
| Drag в Applications + первый запуск             | 30 сек  |
| Bootstrap Python + venv                         | 1-2 мин |
| Скачивание модели (внутренний storage)          | 3-6 мин |
| Permissions Mail + Calendar                     | 30 сек  |
| **Итого до «готово»**                            | **~10 мин** |

10 минут — это **верхняя граница**, при которой пилот ещё имеет шанс
на успех. Если бутстрэп выйдет за 20 минут — пилоты бросят.

## Альтернатива: вообще без локальной установки (если пилот горит)

Если **очень срочно** нужно показать работающий продукт 5 не-инженерам,
и нет 1-2 недели на упаковку — **временно** разверни pa-clean на
**сервере** с доступом через VPN/Tailscale:

- ✅ Тестировщик просто открывает URL в браузере — ничего не ставит.
- ✅ Все обновления централизованные.
- ⚠ Mail/Calendar интеграция **не работает** — это локальные macOS
  приложения тестировщика, а сервер их не видит.
- ⚠ Подходит ТОЛЬКО для тестирования UI / promptов / суммаризации
  на тестовом vault. Реальные Mail/Calendar сценарии остаются для
  локальной установки.

То есть так можно протестировать **WebUI, чат, промпты**, но не сам
смысл pa-clean (синк личной почты). Может пригодиться как
«smoke-test перед полноценным пилотом».

## Резюме

1. **Без подписи Developer ID на корпоративных машинах ничего не
   взлетит** — это блокер №1, заводим certificate сейчас же.
2. **Embedded wheelhouse + внутренний storage для модели** —
   обязательно, если есть малейший шанс что PyPI/HuggingFace заблокированы.
3. **Onboarding UI критичнее самого пакетирования** — без визарда с
   permissions и model-download пилот провалится у 80% тестировщиков.
4. **Self-update через PyApp** — без него цикл «баг → фикс →
   доставка» убивает обратную связь.
5. **Пилот ≠ продакшен.** Стремимся к 10-минутному onboarding-у и
   тесному каналу обратной связи через Slack, а не к идеальной
   App Store-готовой версии.
